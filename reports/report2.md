# Звіт з лабораторної роботи: Побудова CI-конвеєра із використанням GitHub Actions

## 1. Виконання Частини A та B (CI-конвеєр та публікація артефактів)
Було створено файл конфігурації `.github/workflows/ci.yml` для виконання CI-пайплайну на хмарних серверах GitHub (`ubuntu-latest`).

* **Тригери (Events):** Налаштовано запуск при `push` та `pull_request` у гілку `main`, а також ручний запуск `workflow_dispatch` із можливістю вибору конкретного модуля для тестування.
* **Матрична стратегія (Matrix Strategy):** Реалізовано паралельний запуск для трьох аналітичних модулів: `01_quality_check`, `02_research`, `03_viz`. Це дозволило уникнути дублювання коду в конфігурації.
* **Кроки виконання (Steps):**
  * Встановлення середовища (Python 3.11) та залежностей (`pandas`, `jupyter`, `nbconvert`, `matplotlib`).
  * Динамічний запуск модулів: для файлів `.ipynb` використано `jupyter nbconvert --execute`, а для `.py` — системний виклик `python`.
* **CD / Публікація результатів:** Після виконання аналітики всі результати (звіти, логи) зберігаються за допомогою `actions/upload-artifact@v4` для подальшого завантаження та аналізу.
* **Проблема великих даних:** Під час розробки виникла проблема з лімітами GitHub (обмеження файлів до 100 МБ). Основний набір даних `reestrtz28.02.2026.csv` (107 МБ) не міг бути завантажений у репозиторій. Хмарний CI завершувався з помилкою `FileNotFoundError`. Для вирішення цієї проблеми було налаштовано локальний агент.

## 2. Виконання Частини C (Self-hosted runner)
Для забезпечення доступу до повного набору відкритих даних було підключено локальний агент (self-hosted runner) на базі ОС macOS. Створено окремий файл конфігурації `.github/workflows/ci-selfhosted.yml`.

**Вирішені інженерні задачі під час налаштування:**
* **Права доступу (Permissions):** Хмарний екшн `setup-python` намагався створити директорії в `/Users/runner/`, що викликало `Permission denied` на macOS. Проблему вирішено шляхом повного переходу на використання системного інтерпретатора `python3` та `pip3`.
* **Доступ до даних:** Замість завантаження даних з GitHub, використано команду `cp` для прямого копіювання файлу `reestrtz28.02.2026.csv` з локальної директорії користувача безпосередньо в ізольовану робочу папку ранера `_work`.
* **Виправлення робочих директорій:** При запуску Python-скриптів (`02_research.ipynb.py`) виникала проблема з відносними шляхами (`../data/`). У пайплайн було додано команду `cd notebooks` перед запуском скрипта, щоб синхронізувати контекст виконання з Jupyter-ноутбуками.
* **Кодування (Encoding):** Виявлено та виправлено помилку `UnicodeDecodeError`, пов'язану з читанням кириличних символів. Кодування файлів у `pandas` було змінено на сумісне (`utf-8`).

## 3. Порівняльний аналіз (GitHub-hosted vs Self-hosted runners)

| Критерій | GitHub-hosted runner (`ubuntu-latest`) | Self-hosted runner (локальний macOS) |
| :--- | :--- | :--- |
| **Швидкість** | Витрачає додатковий час на запуск віртуальної машини, завантаження репозиторію з нуля та "чисте" встановлення залежностей. | Працює швидше на етапі ініціалізації, оскільки середовище (Python, системні бібліотеки) вже налаштовано. Швидкість виконання залежить виключно від потужності локального процесора. |
| **Доступ до локальних ресурсів** | Суворо обмежений. Не має доступу до локальних баз даних чи файлів понад ліміти Git (100 MB). Вимагає використання "mock" даних або зовнішніх хмарних сховищ (S3). | **Абсолютна перевага.** Має прямий доступ до локальної файлової системи ПК. Дозволив миттєво скопіювати локальний датасет вагою 107 МБ у робочу директорію без порушення правил GitHub. |
| **Ризики (безпека та стабільність)** | Максимально безпечний (кожен запуск ізольований у тимчасовій віртуальній машині). Висока стабільність (сервери GitHub доступні 24/7). | **Ризик безпеки:** шкідливий код у Pull Request може бути виконаний прямо на локальній машині користувача. **Ризик стабільності:** пайплайн не спрацює ("runner offline"), якщо ПК вимкнений, знаходиться в режимі сну (`safe_sleep`) або відсутнє підключення до Інтернету. |

## Висновок
Під час виконання лабораторної роботи було успішно побудовано CI/CD конвеєр. Практично доведено, що для проєктів, які працюють з великими наборами даних (Data Science / AI Analytics), використання self-hosted ранерів є ефективнішим підходом, оскільки дозволяє обійти мережеві обмеження та безпосередньо взаємодіяти з локальними файловими масивами.
